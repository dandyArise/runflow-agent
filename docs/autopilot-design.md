# Autopilot Mode Design

Status: design-only, not approved for runtime.

`autopilot` is a future permissioned workflow for proposing and, only after explicit human approval, applying limited RunFlow actions. It is not part of the current runtime. This document defines the safety contract that must exist before any implementation starts.

## Decision

Autopilot remains blocked until all gates in this document are implemented and reviewed. The first implementation must produce dry-run proposals only. Any mutation path must come later, behind explicit config, command-line confirmation, per-action allowlists, and audit coverage.

## Goals

- Convert watch and oncall evidence into proposed actions a human can review.
- Keep dry-run proposals deterministic and testable.
- Require explicit opt-in before any mutation-capable code path exists.
- Require a per-action allowlist for every possible mutation.
- Require visible audit records for proposals, approvals, denials, and applied actions.
- Make rollback or recovery guidance part of every proposed mutation.

## Non-Goals

- No default autonomy.
- No hidden background loop.
- No mutation without human approval.
- No shell execution.
- No secrets access.
- No outbound network calls by default.
- No notification delivery by default.
- No broad "fix everything" mode.
- No automatic job execution, cancellation, or rerun in the first implementation.

## Dependency Order

Autopilot must build on already bounded modes:

1. `watch`: observes local state and emits snapshots.
2. `oncall`: groups incidents and prepares human handoff evidence.
3. `autopilot plan`: turns evidence into dry-run proposals.
4. `autopilot apply`: future, approval-gated, mutation-capable path.

Only step 3 is eligible for the first runtime implementation.

## Proposed CLI

Design target only:

```powershell
runflow-agent autopilot plan --root . --from-watch .\.flow\agent\watch\latest.json
runflow-agent autopilot plan --root . --from-oncall .\.flow\agent\oncall\handoff.json
runflow-agent autopilot plan --root . --format json
runflow-agent autopilot plan --root . --output .\.flow\agent\autopilot\proposal.json
```

Future mutation path, not approved:

```powershell
runflow-agent autopilot apply --proposal .\.flow\agent\autopilot\proposal.json --confirm <approval-token>
```

`apply` must not exist until the permission model, approval flow, rollback design, and tests are complete.

## Inputs

Local reads only for the first implementation:

- `.flow/jobs`;
- `.flow/runs`;
- `logs/`;
- `.flow/agent/watch/*.json`;
- `.flow/agent/oncall/*.json`;
- `.flow/agent/audit.jsonl`;
- explicit proposal files passed by the user.

No provider access is required for baseline dry-run proposals. If model-assisted proposal generation is added later, it must use strict JSON, deterministic repair, bounded prompts, and the same denial rules as existing agent commands.

## Output Contract

Dry-run JSON should use this shape:

```json
{
  "kind": "autopilot_proposal",
  "root": ".",
  "generated_at": "unix-seconds",
  "mode": "dry_run",
  "source": {
    "watch_snapshot": ".flow/agent/watch/latest.json",
    "oncall_handoff": ".flow/agent/oncall/handoff.json"
  },
  "summary": {
    "proposals": 1,
    "requires_approval": 1,
    "blocked": 0
  },
  "proposals": [
    {
      "id": "proposal-001",
      "action": "review_failed_run",
      "target": "run-123",
      "status": "requires_approval",
      "risk": "low",
      "reason": "run-123 failed at upload",
      "evidence": [
        "oncall severity high",
        "failed_step upload",
        "stderr.log present"
      ],
      "preview": {
        "would_change": [],
        "would_run": []
      },
      "approval": {
        "required": true,
        "token_hint": "generated by future approval command"
      },
      "rollback": [
        "No rollback needed for review-only proposal."
      ]
    }
  ],
  "blocked": []
}
```

Text output should be a compact equivalent of the same data.

## Permission Model

Baseline proposal actions:

- `review_failed_run`: read-only recommendation.
- `inspect_workflow`: read-only recommendation.
- `prepare_handoff`: read-only recommendation.

Mutation actions remain denied until separately approved:

- `run_job`;
- `cancel_run`;
- `rerun_step`;
- `edit_workflow`;
- `write_config`;
- `send_notification`;
- `call_webhook`;
- `execute_shell`.

Every future mutation action must define:

- exact command name;
- allowed targets;
- required approval;
- preview output;
- audit fields;
- rollback or recovery note;
- tests for denied variants.

## Approval Flow

Future approval flow must be explicit:

1. Generate proposal in dry-run mode.
2. Human reviews proposal file.
3. Human runs a separate approval command that emits an approval token.
4. Apply command receives proposal path and approval token.
5. Apply command validates proposal freshness, allowlist, and approval token.
6. Apply command records audit before and after action.

Approval tokens must be local, short-lived, scoped to one proposal, and invalid after use.

## Audit

Dry-run proposal audit:

```json
{
  "command": "autopilot plan",
  "status": "success",
  "changed_files": [],
  "warnings": ["proposal-001 requires approval"]
}
```

Future apply audit must record:

- proposal ID;
- action;
- target;
- approval token fingerprint, never the raw token;
- before status;
- after status;
- changed files;
- result.

## Safety Rules

Hard deny for first implementation:

- mutation;
- shell execution;
- secrets access;
- outbound network calls;
- notification delivery;
- background daemon behavior;
- apply command;
- approval token generation.

Allowed for first implementation:

- bounded local reads;
- dry-run proposal output;
- explicit `--output` writes under user-selected paths;
- audit writes under `.flow/agent/audit.jsonl`.

## Failure Behavior

- Invalid source JSON should fail closed.
- Missing source files should produce a clear error.
- Empty evidence should produce a valid proposal document with no proposals.
- Unknown actions should be blocked, not downgraded.
- Proposal files should be stable and deterministic enough for review.

## Implementation Plan

1. Add `autopilot plan` only.
2. Reuse `watch` and `oncall` output contracts as source inputs.
3. Generate read-only proposals only.
4. Add JSON and text output.
5. Add `--output` and audit writes.
6. Add tests for strict JSON, denied actions, empty evidence, invalid source files, and blocked unknown actions.
7. Do not add `apply`.

## Tests

Required tests before runtime:

- empty sources return a valid dry-run proposal with no actions;
- failed run evidence produces a read-only proposal;
- mutation actions are denied;
- unknown actions are blocked;
- invalid source JSON fails closed;
- `--format json` returns strict JSON;
- `--output` writes only the requested file;
- audit records proposal warnings;
- no shell, webhook, notification, run, cancel, or rerun path exists.

## Open Decisions

- Whether proposal IDs should be deterministic hashes or sequence numbers.
- Whether approval tokens belong in this binary or an external controller.
- Whether model-assisted proposal drafting is worth adding.
- Whether apply should ever be part of this project or remain out of scope.
